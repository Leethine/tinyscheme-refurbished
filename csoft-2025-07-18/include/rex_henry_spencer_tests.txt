#*******************************************************************************
#
#    Modified Henry Spencer's regexp(3) Test File
#
#    This is Henry Spencer's regexp(3) test file modified to use
#    the RE syntax of my REX_UTIL package.
#
#    Each line is a test with 6 tab-separated fields:
#
#        <test#>	- Test number.
#        <find>		- Regular expression.
#        <text>		- Text against which the RE is matched.
#        <status>	- Expected test status:
#			    'c' for RE compile error,
#			    'y' if the RE is matched in <text>,
#			    'n' if the RE is NOT matched in <text>.
#        <source>	- A string whose references to the match and the
#			  capture groups in the regular expression will be
#			  substituted with the matched text.
#        <result>	- Expected result of the substitution.
#
#    (NOTE that <source> is NOT used for a typical search and replace in
#    <text>, but to test substitution.  The source string only makes use
#    of the matched text and not the portions of <text> before and after
#    the match.)
#
#    Henry Spencer's actual test file had 5 fields per line.  To make it
#    easier to identify and correlate tests, I added a new field, the test
#    number, to both (1) my modified tests and (2) the copy of Henry's
#    original tests further down below.
#
#    A solitary hyphen ("-") for a field indicates that the field is
#    not used in the test.  Tests whose REs should fail to compile
#    have a <text> field of "-".  Tests which don't test substitution
#    have <replace> and <result> fields of "-".
#
#    Exactly one tab separates adjacent fields.  Consequently, a tab
#    at the very beginning of the line means <find> is an empty string
#    (""):
#
#        "<test#>\t\t<text>\t<status>\t<replace>\t<result>"
#
#    Likewise, two consecutive tabs mean the in-between field is
#    an empty string; for example, two tabs after the <find> field
#    mean that <text> is an empty string:
#
#        "<test#>\t<find>\t\t<status>\t<replace>\t<result>"
#
#    At the end of this file, you'll find Henry Spencer's original test
#    file (commented out).
#
#*******************************************************************************
#
#    To compile this file into a test program:
#
#    Convert the file into an array of C strings using the following sed
#    commands in the order given:
#
#        s|\\|\\\\|g	-- Escape all backslashes in the file so that the
#			   C compiler inserts a literal backslash in the
#			   string,  "\" => "\\".
#        s|"|\\"|g	-- Escape all double quotes in the file so that
#			   the C compiler can distinguish them from the
#			   double quotes that will begin and end the string.
#        s|	|\\t|g	-- Convert all tabs to C's "\t" specifier.  Either
#        s|\t|\\t|g	   one of these commands will work.  The first has
#			   an actual tab character for the search string;
#			   the second has "\t", which sed will treat as
#			   a tab character in the search string.
#        s|\f|\\f|g	-- I like form feeds, so sue me!
#        s|^|    "|	-- Indent and insert a double quote delimiter at
#			   the beginning of the string.
#        s|$|",|	-- Append a double quote delimiter at the end of
#			   the string, plus a comma.
#
#    Insert the output into a C array of strings:
#
#        static  const  char  *HenrySpencersTestFile[] = {
#        "<first-line>",
#        ...,
#        "#        (2) Replaced all \"\\N\"s in 4th and 5th columns by \"$N\".",
#        ...,
#        "#65\t((a))\tabc\ty\t&-\\1-\\2\ta-a-a",
#        ...,
#        "<last-line>",
#        NULL
#        } ;
#
#    The comment line "(2)" is the escaped translation of this line in this
#    file:
#
#        ==>#        (2) Replaced all "\N"s in 4th and 5th columns by "$N".<==
#
#    The other line shown is the translation of one of Henry Spencer's tests;
#    the 5 regions of space are the 5 tab characters separating the 6 fields:
#
#        ==>#65	((a))	abc	y	&-\1-\2	a-a-a<==
#
#*******************************************************************************

#        Wow!  I had tested my regular expression package against Henry
#        Spencer's test file--with appropriate changes made for my RE
#        syntax--back in the early 1990s.  Somehow, my modified test
#        file disappeared.  When I downloaded Henry Spencer's rexexp(3)

#        ("../tools/trex.c"), I thought I was going to have to again go
#        through his test file, line by line and character by character,
#        to convert it to my RE sytax.
#
#        Then, on May 27, 2019 at about 12:40 A.M., I happened upon my
#        heretofore missing test file in a long-forgotten subdirectory.
#        Hooray!!!  A big load off my mind!  (Back in 1996, there was
#        a discussion about regular expressions on the comp.lang.tcl
#        USENET newsgroup and some kind soul had recommended my package.
#        I assembled a mini-distribution of my LIBGPL library containing
#        only the files needed by the REX_UTIL package and my test program
#        that read and checked the modified test file.  The Tcl developers
#        went ahead and used Henry Spencer's package anyway, but at least
#        I now have the test file AND the test program that read the file!)
#
#    Here is Henry Spencer's regexp(3) test file modified to use the RE
#    syntax of my REX_UTIL package.  My original REX_UTIL package used
#    a different syntax, so the modifications were more extensive back
#    in the 1990s.  In 2021, after experience with the Sigil EPUB editor's
#    regular expressions, I changed the syntax to more closely match that
#    used by Henry Spencer.  Consequently, the only modification required
#    now was:
#
#        (1) Replaced all "&"s in the 4th column by "\&".
#
#    Henry used "&" to refer to the full matched text and "\N" to refer
#    to the numbered, parenthesized capture groups.  I now use "\&" and
#    "\N", respectively.
#
#    Test #122 is the character 'a' enclosed in 10 sets of nested parentheses.
#    Henry's rexexp(3) code only allows 9 capture groups, \1 through \9, as
#    did my original REX_UTIL package.  Consequently, the test is flagged as
#    producing an RE compilation error ('c').  Since REX_UTIL now allows an
#    unlimited number of capture groups (within reason), I changed the test
#    to pass instead of reporting an error because the RE failed to fail to
#    compile.
#
